# 测试结果总结报告

## 执行日期
2025-02-26

## 测试环境
- 硬件：Raspberry Pi 5 Model B Rev 1.1
- OS：Ubuntu 24.04 LTS (Noble)
- GPIO 接口：/dev/gpiochip4 (RP1 controller)
- 测试方法：libgpiod C++ edge events

## Stage 1 测试结果摘要

| 频率 | CH1 | CH2 | CH3 (GPIO23) | CH4 | 状态 | 备注 |
|------|-----|-----|--------------|-----|------|------|
| 10 kHz | <0.01% | <0.01% | <0.01% | <0.01% | ✅ PASS | 所有通道优秀 |
| 20 kHz | 0.01% | 0.06% | 0.37% | 0.19% | ✅ PASS | CH3最高但<1% |
| 30 kHz | 0.06% | 0.46% | 0.12% | 0.16% | ✅ PASS | CH2最高但<1% |

| 50 kHz | - | - | - | - | ❌ FAIL | 边沿丢失>5% |

## 关键发现

### 1. 频率边界
- **通过边界**：30 kHz（所有通道 <5%）
- **失败边界**：40 kHz（CH3 失败）
- **实际可用范围**：10-30 kHz（稳定）

### 2. 通道性能不均
- **CH1 (GPIO5)**：性能稳定，40 kHz 时 4.93%
- **CH2 (GPIO6)**：性能稳定，40 kHz 时 4.65%
- **CH3 (GPIO23)**：性能最差，40 kHz 时 10.78% ❌
- **CH4 (GPIO24)**：性能最好，40 kHz 时 3.11%

### 3. 性能下降模式
- 10 kHz → 20 kHz → 30 kHz：性能下降缓慢（<1%）
- 30 kHz → 40 kHz：性能急剧下降（CH3 从 0.12% 到 10.78%）

### 边沿丢失模式
- **CH3 的丢失率是 CH4 的 3.5 倍**
- **CH3 的实测频率比 CH4 低约 8%**
- **CH3 的上升沿和下降沿都明显偏少**

## 问题分析

### 可能原因（按可能性排序）

#### 1. GPIO23 硬件问题 ⭐⭐⭐⭐⭐
- 物理引脚接触不良
- 连接线质量差
- 信号干扰（Pin 16 位于中间位置）

#### 2. 内核调度问题 ⭐⭐⭐⭐
- GPIO23 由负载更高的 CPU 核心处理
- 中断处理优先级不均
- libgpiod 缓冲区配置问题

#### 3. 多通道竞争 ⭐⭐⭐
- 4 通道同时采集导致资源竞争
- GPIO23 可能是最后被处理的

#### 4. 引脚布局/干扰 ⭐⭐
- Pin 16 物理位置特殊
- 可能受到邻近引脚干扰

## 下一步建议

### 优先级 1：单独测试 GPIO23 ⏱️ 15分钟

**目的**：排除多通道竞争，确认是否硬件问题

**步骤**：
1. 修改 `freq_measure_cpp.cpp`，使用 GPIO23 (line_offset = 23)
2. 编译：`./compile_freq_cpp.sh`
3. 测试 40 kHz
4. 对比 GPIO5 的单独测试结果

**判断标准**：
- 如果 GPIO23 单独测试 <5% → 多通道竞争问题
- 如果 GPIO23 单独测试 >5% → 硬件问题

### 优先级 2：测试 35 kHz ⏱️ 5分钟

**目的**：细化通过/失败边界

**步骤**：
```bash
sudo ./multi_channel_freq_test 3
```

**判断标准**：
- 如果 35 kHz 通过 → 边界在 35-40 kHz
- 如果 35 kHz 失败 → 边界在 30-35 kHz

### 优先级 3：检查中断分布 ⏱️ 10分钟

**目的**：确认 GPIO23 的中断处理情况

**步骤**：
1. 启动中断监控：`watch -n 0.1 'cat /proc/interrupts | grep -i gpio'`
2. 运行 40 kHz 测试
3. 观察哪个 CPU 核心处理 GPIO23 的中断

### 优先级 4：根据优先级 1 结果决策

#### 场景 A：GPIO23 单独测试正常（<5%）
- **结论**：多通道竞争问题
- **解决方案**：
  - 优化系统调度参数
  - 使用 CPU 亲和性
  - 减少通道数量（4→2）

#### 场景 B：GPIO23 单独测试失败（>5%）
- **结论**：硬件或驱动问题
- **解决方案**：
  - 更换 GPIO23 为其他引脚（如 GPIO17/18/22/27）
  - 更新配置文件
  - 重新测试
